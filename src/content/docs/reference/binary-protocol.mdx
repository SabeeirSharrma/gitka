title: Binary Protocol
description: Specification of the binary protocol used for TCP communication between Transit and language bridges.

Transit uses a simple binary protocol for communication between the Node.js host and language-specific bridge processes (Java, Python, etc.) over TCP on localhost.

## Packet Structure

Each packet consists of a fixed 10‑byte header followed by a variable‑length payload.

| Offset | Size (bytes) | Description                     |
|--------|--------------|---------------------------------|
| 0      | 1            | Protocol version (currently `1`)|
| 1      | 1            | Packet type                     |
| 2      | 4            | Request ID (little‑endian uint32)|
| 6      | 4            | Payload length (little‑endian uint32)|
| 10     | *n*          | Payload (UTF‑8 JSON string)     |

### Packet Types

| Value | Meaning                | Direction   |
|-------|------------------------|-------------|
| `0`   | `CALL_REQUEST`         | Host → Bridge|
| `1`   | `CALL_RESPONSE`        | Bridge → Host|
| `2`   | `HEALTH_PING`          | Host → Bridge|
| `3`   | `HEALTH_PONG`          | Bridge → Host|

## Payload Format

The payload is a UTF‑8 encoded JSON string.

### CALL_REQUEST

```json
{
  "functionName": "string",
  "arguments": [ /* JSON‑serializable values */ ]
}
```

### CALL_RESPONSE

```json
{
  "status": "ok" | "error",
  "result": any,          // present when status === "ok"
  "error": "string"       // present when status === "error"
}
```

### HEALTH_PING / HEALTH_PONG

Both messages have an empty payload (`{}`) or may contain a timestamp; the bridge simply replies with a PONG.

## Example Exchange

1. **Host → Bridge** (`CALL_REQUEST`)
   - Version: `1`
   - Type: `0`
   - Request ID: `1234`
   - Payload length: `n`
   - Payload: `{"functionName":"processData","arguments":[{"foo":"bar"}]}`

2. **Bridge → Host** (`CALL_RESPONSE`)
   - Version: `1`
   - Type: `1`
   - Request ID: `1234`
   - Payload length: `m`
   - Payload: `{"status":"ok","result":{"ok":true}}`

## Error Handling

If the bridge encounters an error while executing the requested function, it returns a `CALL_RESPONSE` with `status: "error"` and an `error` field containing a human‑readable message.

## Connection Management

- The host initiates a TCP connection to the bridge on an ephemeral port provided by the bridge’s stdout (`PORT=<port>`).
- The connection remains open for the lifetime of the language handle.
- The host sends periodic `HEALTH_PING` messages to detect a dead bridge; missing `HEALTH_PONG` responses trigger reconnection logic.

## Security

The protocol is intended for loop‑back communication only (`127.0.0.1`). No encryption or authentication is performed because the traffic never leaves the host machine.
