# Advanced Features

## Volume Splitting

Split archives into fixed-size parts. Useful for burning to CD/DVD or working around FAT32's 4 GB file size limit.

```bash
# Enable during init (4 GB parts for FAT32)
gitka init --target /mnt/usb --volume-size 4096

# Or enable via config
gitka config --set compression.volume_splitting.size_mb=700  # CD-sized parts
gitka config --set compression.volume_splitting.size_mb=4096  # FAT32 limit
gitka config --set compression.volume_splitting.size_mb=off   # disable
```

When enabled, archives are split across multiple files:
```
repos/archive/my-project.gitka.zst      # Part 1 (primary)
repos/archive/my-project.gitka.zst.002  # Part 2
repos/archive/my-project.gitka.zst.003  # Part 3 (if needed)
```

All operations automatically detect and handle multi-volume archives.

## Cross-Repo Deduplication

When multiple repos share common files (e.g., shared libraries, node_modules), dedup saves space by storing each unique content block only once.

```bash
# Enable during init
gitka init --target /mnt/usb --dedup

# Or enable/disable via config
gitka config --set compression.dedup=true
gitka config --set compression.dedup=false
```

Dedup hashes each file's content with SHA-256 before compression. If the content already exists, only a reference is written. Dedup usage is tracked per-operation via `record_saved_bytes()`.

## Encryption

Gitka supports optional **AES-256-GCM encryption** for archives. When enabled:

- A random salt is generated and persisted to the config file
- Each archive is encrypted with a key derived from the salt
- The salt must remain consistent — Gitka auto-manages it via `ensure_encryption_salt()`

```bash
# Enable encryption
gitka config --set toggles.encryption=true

# Disable encryption
gitka config --set toggles.encryption=false
```

## Recovery Records

Optional **par2 recovery records** provide corruption protection. When enabled:

- Each archive gets ~25% overhead for redundancy records
- `gitka repair <repo>` can recover from partial corruption
- Budget checks account for the recovery overhead automatically

```bash
# Enable recovery records
gitka config --set toggles.recovery_records=true

# Repair a corrupted repo
gitka repair <repo>
```

## Host-Side Extraction

By default, Gitka extracts repos to the USB drive. With host-side extraction, repos are extracted to the host computer's temp directory instead — significantly faster for I/O-bound work.

```bash
# Enable host-side extraction
gitka config --set extraction.target=host

# Revert to USB extraction
gitka config --set extraction.target=usb
```

When enabled, the budget check uses the host machine's free disk space instead of the USB drive's free space.

## Smart Sync & Auto-Merge

`gitka sync` performs intelligent synchronization:

1. **Default branch detection** — reads `origin/HEAD` to determine the repo's default branch, falling back to `main`/`master`/`trunk`
2. **Auto-merge** — when local and remote have diverged with no conflicts, Gitka creates a proper merge commit with two parents
3. **Incremental sync** — only fetches repos that have changed since last sync
4. **Recovery overhead estimation** — budget checks account for ~25% par2 overhead when recovery records are enabled

```bash
gitka sync           # auto-merges clean divergences
gitka sync --json    # JSON output for scripting
```

## Audit Trail

When you lock a repo after offline work, Gitka shows a detailed audit trail:

- Number of commits made
- Files touched/modified
- Session duration
- Archive size before and after

## JSON Output

All commands support `--json` for scripting and CI integration:

```bash
gitka sync --json
gitka status --json
gitka scan --json
```

## Auto-Update

Gitka includes a built-in update system that keeps both CLI and GUI up to date:

```bash
# Check for updates without installing
gitka update --check

# Update both CLI and GUI to the latest version
gitka update

# Update CLI only (skip GUI build)
gitka update --no-gui

# JSON output for scripting
gitka update --json
```

The update command:
1. Checks GitHub releases for the latest version
2. Compares with your current version
3. Clones the latest source and rebuilds both CLI and GUI
4. Installs the new binaries in-place

The install scripts (`install.sh` and `install-windows.ps1`) also build both CLI and GUI by default. Use `--cli-only` / `-CliOnly` to skip the GUI build during installation.
